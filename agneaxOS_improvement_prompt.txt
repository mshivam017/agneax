HIGH LEVEL ANALYSIS

Agneax OS currently looks like a strong prototype or early MVP, not yet a production-ready OS.
The UI bridge structure is good for quick development, and the Rust daemon direction is correct.
But many important system features are still mocked or unsafe for real OS use.

The biggest improvement need is to convert the project from "demo shell with simulated backend"
into "real bootable OS with safe system services, real installer, tested package management,
polished UX, and reliable build pipeline."

CURRENT STRENGTHS

- Clean project structure.
- Separate modules for desktop, store, installer, control center, configs, tests, and docs.
- Rust daemon is a good choice for system-level services.
- QML/PySide6 allows fast UI iteration.
- C++ tiling helper has simple and testable geometry logic.
- GitHub Actions workflow exists for ISO build.
- Basic Python tests are present.
- Build script uses debootstrap and squashfs, which is a reasonable Linux ISO approach.

IMPORTANT PROBLEMS FOUND

1. README and docs have encoding corruption
- Symbols and emojis are displayed as broken characters like "ðŸ".
- This makes the project look unprofessional.
- Fix by saving docs as UTF-8 and removing or replacing broken characters.

2. Installer is currently simulated
- installer/main.py uses hardcoded drives like /dev/sda and /dev/nvme0n1.
- Installation steps are only progress messages.
- It does not actually partition, format, copy rootfs, install bootloader, or create a real user.
- This is dangerous if users think it is production-ready.

3. Store does not really install selected apps
- store/main.py has a hardcoded catalog.
- installApp only simulates progress and calls pkg_update_cache.
- It does not install the selected package ID.
- It creates .desktop launchers that point to /opt/agneax/{app_id}/main.py, which will not exist for normal apps like Firefox or VLC.

4. Privileged command handling is unsafe
- packages/agneax-core/src/main.rs uses Command::new("sh").arg("-c").arg(cmd).
- Even though current commands are fixed strings, this pattern should not be used for an OS daemon.
- System actions should use allowlisted commands with explicit arguments, polkit, DBus, or dedicated safe handlers.

5. No systemd service for agneax-core found
- The build script copies agneax-core into /usr/bin.
- But no visible systemd unit enables and starts it.
- Desktop and control center expect daemon at 127.0.0.1:9090, but the daemon may not be running in the live ISO.

6. Desktop launcher uses os.system
- desktop/main.py uses os.system for launching apps.
- Generic app_name is passed into shell command.
- Replace with subprocess.Popen using a strict allowlist of app IDs and command arrays.

7. Build scripts need hardening
- scripts/install_deps.sh appears to end with an extra EOF line.
- scripts/build_iso.sh uses rm -rf "$WORKDIR" as root. It should validate WORKDIR before deleting.
- Chroot mounts need a trap cleanup handler so failed builds do not leave mounts behind.
- PySide6 is installed with pip --break-system-packages inside rootfs. Prefer Debian packages or a controlled venv/package.

8. Claims are ahead of implementation
- README says production-ready, secure by default, app store, installer, terminal, file manager, gaming ready.
- Actual code is still largely mock/demo.
- Update wording to "prototype", "alpha", or implement the missing real functionality.

9. Tests are too shallow
- Current tests validate mock JSON and fallback calculations.
- Missing tests for Rust daemon RPC, invalid input, installer safety, store package mapping, build scripts, and QML loading.

10. UX consistency needs work
- QML files should share a design token file for colors, spacing, typography, shadows, and animation durations.
- Windows/macOS-like ease comes from consistency, not only visual style.
